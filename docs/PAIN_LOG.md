## 2026-05-05 — Прозрачный overlay: фон сквозь дим легко спутать с собственным контентом

**Симптом:** На скриншоте slide 3 тутора в средней области видны
цветные прямоугольники с подписями «MEGA QUADS», «KICKER BLITZ»
и т.п. — они напоминали мини-плитки slide 4. Заподозрил баг
видимости (slide 4 рендерится поверх slide 3). Полчаса искал
источник в коде. Оказалось: backdrop overlay'я имел `alpha = 0.78`,
лобби сзади просвечивало, а его machine cards имели те же
названия что и мои `tutor.tile.*` ключи (контент скопирован
из figma → совпадение). Никакого бага в коде overlay не было.

**Корневая причина:** При `alpha < 1` любой UI позади overlay'я виден.
При визуальной отладке легко принять подсвеченный фон за свой
контент, особенно если контент совпадает текстом или цветом.

**Правило:** При подозрении на «лишние элементы» в overlay'е с
`alpha < 1` — первым делом временно поставить `bg.alpha = 1.0`
(или ColorRect полностью чёрный поверх backdrop'а). Если
предполагаемый «баг» исчез — это был фон, не свой контент.
Только после этого искать баг в коде. Не тратить время на
теории про порядок рендера, видимость, modulate каскады, пока
не отделил «своё» от «чужого».

## 2026-05-05 — Label gradient через canvas_item шейдер: `UV` это атлас глифа, не Control rect

**Симптом:** Сделал шейдер для двухцветного вертикального градиента
на тексте: `COLOR.rgb = mix(c_top, c_bottom, UV.y) * tex.rgb`.
Получил мусор: каждая буква имела собственный полный gradient
(top→bottom), не единый поток через весь Label.

**Корневая причина:** В Godot 4 `Label` рендерит каждый глиф как
отдельный quad из glyph-atlas текстуры. `UV` во fragment-шейдере —
координаты внутри атласа, не локальные координаты Control'а.
`UV.y == 0` для верха конкретного глифа в атласе, не для верха
Label rect'а.

**Правило:** Для label-wide градиента использовать `varying vec2
v_local` плюс vertex-этап:
```glsl
varying vec2 v_local;
void vertex() { v_local = VERTEX; }   // local Control coords (0..size)
void fragment() {
    float t = clamp(v_local.y / max(rect_height, 1.0), 0.0, 1.0);
    vec3 grad = mix(c_top.rgb, c_bottom.rgb, t);
    COLOR = vec4(grad, texture(TEXTURE, UV).a * COLOR.a);
}
```
+ `uniform float rect_height` пересылается из GDScript на сигнале
`Label.resized` (учитывает word-wrap). См.
`shaders/text_gradient.gdshader` + `_make_grad_label()` в
`scripts/tutorial_overlay.gd`.

Аналогичный паттерн нужен для любых других сценариев когда
canvas_item шейдер должен «видеть» Control-целое (не атлас):
TextureRect с EXPAND, Button с stylebox PNG. UV там тоже
текстурный, не Control'овый.

## 2026-05-05 — Theme-specific game scripts параллельны базовым, хук в одном не покрывает другое

**Симптом:** Добавил `SaveManager.record_machine_win(...)` в
`scripts/game.gd._on_hand_evaluated`. На multi-hand и spin сработало;
на supercell single-hand — нет. Игрок выбил комбо, статистика осталась
пустой.

**Корневая причина:** Supercell single-hand использует
`scripts/themes/supercell/game_supercell.gd` (extends Control), который
дублирует логику game.gd, а не наследуется от него. Multi-hand и Spin
тематические скрипты, наоборот, наследуют базу через
`extends "res://scripts/multi_hand_game.gd"` / `spin_poker_game.gd`,
поэтому хуки в базе работают для них автоматически.

**Правило:** При добавлении логики в `_on_hand_evaluated` /
`_on_hands_evaluated` / `_on_lines_evaluated` (или любой другой
обработчик сигнала менеджера) проверь все три класса:
- `scripts/game.gd` (классическая база single-hand)
- `scripts/themes/<theme>/game_<theme>.gd` (тематические single-hand,
  могут НЕ наследоваться от game.gd)
- `scripts/multi_hand_game.gd` + `scripts/spin_poker_game.gd`
  (тематические подклассы их наследуют)

Перед коммитом: `grep -rn "func _on_<event>" scripts/`. Больше одной
реализации — обнови все.

## 2026-05-05 — HBox/VBox с EXPAND_FILL внутри ScrollContainer растягивает детей вопреки custom_minimum_size

**Симптом:** Карточки магазина (`scripts/shop_overlay.gd._build_pack_card`,
`custom_minimum_size = Vector2(240, 460)`) рендерились высотой ~800px
вместо 460. Картинка фишек болталась в центре с большими пустыми
зонами. Баг жил давно, заметили случайно.

**Корневая причина:** `HBoxContainer row` внутри `ScrollContainer`
имел `size_flags_vertical = EXPAND_FILL`. Скролл занимал всю высоту
`_overlay`, row растягивался по нему, дети HBox наследовали row.size.y
(Control default = SIZE_FILL), игнорируя свой `custom_minimum_size.y`.

**Правило:** Любой ребёнок `H/VBoxContainer`, у которого declaring
fixed size в перпендикулярной оси, должен явно объявить
`size_flags_<perp_axis> = SIZE_SHRINK_CENTER` (или _BEGIN/_END), если
сам контейнер имеет `EXPAND_FILL` по той оси.
custom_minimum_size — это нижний пол, не верхний потолок.

## 2026-05-05 — Async layout race conditions: координаты могут быть «переписаны» соседней корутиной

**Симптом:** В Classic single-hand карты после первого DEAL внезапно
сжимались на половину; layout «магически чинился» только после ручного
ресайза окна редактора. До этого пять итераций фиксов не помогали.
Build 5 → 11 шёл с этой проблемой в разных формах.

**Корневая причина:** Две корутины, обе стартующие из `_ready` через
`call_deferred` / `await get_tree().process_frame`, обе пишут в одни
и те же свойства одного Control:
- `_layout_middle()` писало `_middle_section.offset_top` /
  `offset_bottom`.
- `_play_entrance_animation()` твинило `_middle_section.position:y`.

Установка `Control.position.y` в Godot **перезаписывает оба
offset'а** (для anchor=0/0 на оси Y). Tween анимации каждый кадр
писал position, обнуляя оффсеты, выставленные `_layout_middle`.
Чтение `position.y` для следующего recompute давало значения из
середины tween'а (off-screen), что приводило к гигантскому
MiddleSection и обрезанному paytable.

Усугублял: листенер `_on_section_resized` имел флаг
`_layout_middle_pending`, который **проглатывал последнее
(правильное)** resized-событие. Paytable догружался за несколько
кадров и слал серию resized; обрабатывалось только первое (со
стейлевыми размерами), последующие игнорировались как «pending».

**Правило для будущего:**

1. **Не читай `position.y` для layout-вычислений**, если что-либо
   ещё может его транзиентно изменять (анимации, tween'ы, focus
   transitions). Читай **settled-значения** аналитически:
   `safe_top + top_section.size.y` и
   `total_h - bot_section.size.y - safe_bottom` —
   эти значения зависят только от размеров и safe-area, не от
   текущего состояния позиции.

2. **Не throttle layout-обновления флагом-замком**. Если несколько
   `resized` событий приходят пока `_layout_middle` ещё в `await`,
   они должны накапливаться или последнее должно выиграть, но
   пропускать их нельзя — финальное (с правильными размерами)
   часто приходит ПОСЛЕ первого (со стейлевыми). Если throttling
   нужен, используй deferred-flush паттерн (последнее событие
   выигрывает), не skip-current.

3. **При нескольких асинхронных функциях из `_ready`, пишущих в
   один Control** — гонка гарантирована. Либо последовательность
   через `await` между ними, либо **разделение ответственности**
   (одна функция трогает offsets, другая — modulate/alpha, не
   position).

4. **Editor preview может скрыть этот класс багов** — тестируй на
   устройстве. У нас баг проявлялся только на iPhone, в редакторе
   layout «волшебно» работал в большинстве случаев.

5. **После cross-cutting layout-фикса в одной сцене** — пройдись
   грепом по остальным игровым сценам (multi_hand_game.gd,
   themes/supercell/*.gd, spin_poker_game.gd) на тот же паттерн.
   В нашем случае повезло — конкретно эта race condition была
   только в Classic single-hand, потому что только там секции —
   siblings корня, а не дети `VBoxContainer`. Но магические числа
   (`side_m = 160`) могли бы повториться в нескольких местах.

См. также `docs/UI_LAYOUTS.md` раздел «Layout архитектура» — там
описано, почему sibling-of-root паттерн принципиально хрупче, чем
parent-Container. Новые сцены делать через VBoxContainer-родителя.

## 2026-05-04 — Safe-area: letterbox ≠ notch, и якорь=1 нужно двигать симметрично

**Симптом 1 (build 3 → 4):** В iOS landscape по краям лобби и игровых
сцен видны тёмно-синие полосы. Background ColorRect размечен FULL_RECT,
визуально кажется full-bleed. На скриншотах Supercell single-hand
полосы остались даже после первого фикса safe-area.

**Корневая причина 1:** Это не safe-area inset — это **letterbox** от
`window/stretch/aspect = "keep_height"`. Базовый viewport `1476×680`
(aspect 2.17), iPhone 14 Pro в landscape ~2.12 — окно шире вьюпорта,
Godot оставляет полосы, в которых виден `RenderingServer` clear color.
FULL_RECT ColorRect покрывает только viewport, не window.

**Фикс 1:** `aspect = "expand"` в `project.godot`. Viewport == window,
полосы исчезают.

---

**Симптом 2 (build 4 → 5):** В Classic single-hand после safe-area
inset «TOTAL BET» налезает на нижний край карт, кнопки прижаты впритык
к нижнему краю.

**Корневая причина 2:** В `safe_area_manager.gd._apply_to` бывшая
правка двигала только нижнее ребро bottom-anchored контейнера
(`anchor_bottom == 1` → `offset_bottom -= db`). Но у `BottomSection`
**оба** ребра привязаны к низу (`anchor_top = 1`, `anchor_bottom = 1`,
`grow_vertical = 0` — растёт ВВЕРХ от дна). Сдвиг только нижнего ребра
сжимает контейнер на `db` пикселей сверху вниз → дочерний VBox с
`TOTAL BET` + кнопками + padding получает меньше высоты, элементы
налезают друг на друга и на карты сверху.

**Фикс 2:** Симметричное правило в `_apply_to` — когда оба ребра
оси привязаны к одному анкору, нужно сдвигать оба:
- `anchor_top == 1` → `offset_top -= db` *(в дополнение к bottom)*
- `anchor_left == 1` → `offset_left -= dr`

**Правило для будущего:**

1. **Жалоба «полосы по краям» в одинаковом месте на всех сценах** →
   letterbox, чини aspect.
2. **Жалоба «контент сжимается / налезает после safe-area фикса»** →
   проверь, не bottom/right-anchored ли контейнер; не сжимаешь ли его
   вместо смещения.
3. **Жалоба «затемнение popup'а не дотягивается до края»** →
   а) до фикса aspect — letterbox; б) после — popup инсетится
   (попал в `_tracked` SafeAreaManager). Проверь, что popup
   добавляется как dynamic child после `_make_full_rect`.

Полный справочник — `docs/SAFE_AREA.md`. Подгружай его при любой
работе с layout / safe-area / новыми popup'ами.

## 2026-04-30 — Supercell переопределяет балансовый путь — звук нужно добавить отдельно

**Симптом:** Звук `balance_increment` добавлен в `game.gd`, слышен в обычной теме,
но не слышен в Supercell single hand.

**Корневая причина:** `game_supercell.gd` полностью переопределяет `_on_credits_changed`
и `_animate_win_increment`. Код `game.gd` там не выполняется вообще. Путь баланса
в Supercell: `credits_changed` → `_on_credits_changed` (своя) → `_run_balance_roll_up`.

**Правило:** При добавлении любого звука на игровое событие — проверять ОБА файла:
`game.gd` и `game_supercell.gd`. Звук баланса (`balance_increment`) в Supercell
должен быть в `_run_balance_roll_up`, не в `_animate_win_increment` — потому что
момент начисления для игрока = приземление фишек, а не старт счётчика выигрыша.

## 2026-04-30 — Не угадывай координаты, измеряй

**Симптом:** Несколько итераций «правильного» способа центрирования
UI-элемента (бейджа над мини-рукой), ни одна не подтверждена в runtime.
Каждая новая попытка строилась поверх непроверенной гипотезы предыдущей.

**Корневая причина:** Подмена измерения предположением. После первой
неудачи продолжал предлагать «более правильный» источник координат
(HBox center → формула → first/last child rect), не проверив, что
реально возвращают эти источники в момент вызова.

**Правило:** Если правка по координатам / позиционированию / layout
не сработала с первого раза — следующий шаг не код, а диагностика.
Добавить временные `print` (или попросить пользователя их запустить),
вывести фактические rect / position / size / pivot. Гипотезы можно
выдвигать только поверх измеренных чисел.

## 2026-04-30 — `get_global_rect()` во время transform-анимации лжёт

**Симптом:** Боковые paytable-бейджи в multi-hand привязывались к
центру экрана при переключении hand_count. Ломались режимы, которые
обещал не трогать.

**Корневая причина:** `_build_paytable_badges()` → `_position_badges()`
читал `_extra_grid.get_global_rect()` ДО окончания tw_in анимации
(grid.scale = 0.15). Считывался уменьшенный rect — бейджи цеплялись
к нему.

**Правило:** Если планируется чтение `get_global_rect()` /
`get_global_position()` после программной transform-анимации
(scale/skew/rotation родителя) — `await tween.finished` ОБЯЗАТЕЛЕН
до чтения. Иначе геометрия отражает промежуточный кадр анимации,
а не финальный.

## 2026-04-30 — `queue_free` контейнера ≠ обнуление ссылок на детей

**Симптом:** Краш «Object previously freed» в `_pulse_winning_badges`
после переключения hand_count с 25 на 100.

**Корневая причина:** `_clear_paytable_badges()` делал `queue_free`
для VBox-стопок, что убивало их Label-детей. Но массив
`_badge_labels[]` оставался с прежними указателями — на следующем
DRAW цикл по нему дёргал мёртвые ноды.

**Правило:** Если ноду queue_free'ят, и её дети где-то записаны
в массивы / словари / переменные — `clear()` этих коллекций
обязателен в той же функции. Godot не зануляет такие ссылки сам.
Пройтись по всем кэшам, ссылающимся на детей удаляемой ноды.

## 2026-04-30 — Контейнер-стопка должен быть фиксированным

**Симптом:** При появлении новых бейджей в накопительной стопке
старые бейджи «перелетают» в новые координаты — визуальный шум.

**Корневая причина:** Контейнер имел `size.y = combined_minimum_size`
и `position.y = bottom_y - size.y`. Каждый новый ребёнок изменял
combined_min — позиция контейнера пересчитывалась — глобальные
координаты старых детей внутри ехали.

**Правило:** Для растущих сверху-вниз / снизу-вверх стопок —
контейнер должен быть фиксированной высоты на всю свободную зону,
а порядок задаётся через `BoxContainer.alignment` (BEGIN/END/CENTER)
+ `move_child(_, 0)` для вставки сверху. Контейнер не двигается —
двигается только free space внутри него, дети остаются на месте.

## 2026-04-30 — «Только для одного режима» через общую функцию ломает все режимы

**Симптом:** Добавление 100-hand layout сломало переключение между
3/5/10/12/25 режимами, хотя обещал «не трогать другие лейауты».

**Корневая причина:** Под флагом нового режима менял общую функцию
`_switch_hand_count` (заменил `_update_paytable_badges` на
`_build_paytable_badges`), что повлияло на все режимы, не только
на новый.

**Правило:** Когда задача звучит «не затронуть остальные режимы» —
любая правка в общей функции должна быть branch'ем под условие
нового режима, а не заменой общего пути. Default-ветка остаётся
байт-в-байт прежней. Если без изменения общего пути нельзя — это
повод явно обсудить с пользователем, не делать молча.
