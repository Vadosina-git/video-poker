## 2026-05-04 — Safe-area: letterbox ≠ notch, и якорь=1 нужно двигать симметрично

**Симптом 1 (build 3 → 4):** В iOS landscape по краям лобби и игровых
сцен видны тёмно-синие полосы. Background ColorRect размечен FULL_RECT,
визуально кажется full-bleed. На скриншотах Supercell single-hand
полосы остались даже после первого фикса safe-area.

**Корневая причина 1:** Это не safe-area inset — это **letterbox** от
`window/stretch/aspect = "keep_height"`. Базовый viewport `1476×680`
(aspect 2.17), iPhone 14 Pro в landscape ~2.12 — окно шире вьюпорта,
Godot оставляет полосы, в которых виден `RenderingServer` clear color.
FULL_RECT ColorRect покрывает только viewport, не window.

**Фикс 1:** `aspect = "expand"` в `project.godot`. Viewport == window,
полосы исчезают.

---

**Симптом 2 (build 4 → 5):** В Classic single-hand после safe-area
inset «TOTAL BET» налезает на нижний край карт, кнопки прижаты впритык
к нижнему краю.

**Корневая причина 2:** В `safe_area_manager.gd._apply_to` бывшая
правка двигала только нижнее ребро bottom-anchored контейнера
(`anchor_bottom == 1` → `offset_bottom -= db`). Но у `BottomSection`
**оба** ребра привязаны к низу (`anchor_top = 1`, `anchor_bottom = 1`,
`grow_vertical = 0` — растёт ВВЕРХ от дна). Сдвиг только нижнего ребра
сжимает контейнер на `db` пикселей сверху вниз → дочерний VBox с
`TOTAL BET` + кнопками + padding получает меньше высоты, элементы
налезают друг на друга и на карты сверху.

**Фикс 2:** Симметричное правило в `_apply_to` — когда оба ребра
оси привязаны к одному анкору, нужно сдвигать оба:
- `anchor_top == 1` → `offset_top -= db` *(в дополнение к bottom)*
- `anchor_left == 1` → `offset_left -= dr`

**Правило для будущего:**

1. **Жалоба «полосы по краям» в одинаковом месте на всех сценах** →
   letterbox, чини aspect.
2. **Жалоба «контент сжимается / налезает после safe-area фикса»** →
   проверь, не bottom/right-anchored ли контейнер; не сжимаешь ли его
   вместо смещения.
3. **Жалоба «затемнение popup'а не дотягивается до края»** →
   а) до фикса aspect — letterbox; б) после — popup инсетится
   (попал в `_tracked` SafeAreaManager). Проверь, что popup
   добавляется как dynamic child после `_make_full_rect`.

Полный справочник — `docs/SAFE_AREA.md`. Подгружай его при любой
работе с layout / safe-area / новыми popup'ами.

## 2026-04-30 — Supercell переопределяет балансовый путь — звук нужно добавить отдельно

**Симптом:** Звук `balance_increment` добавлен в `game.gd`, слышен в обычной теме,
но не слышен в Supercell single hand.

**Корневая причина:** `game_supercell.gd` полностью переопределяет `_on_credits_changed`
и `_animate_win_increment`. Код `game.gd` там не выполняется вообще. Путь баланса
в Supercell: `credits_changed` → `_on_credits_changed` (своя) → `_run_balance_roll_up`.

**Правило:** При добавлении любого звука на игровое событие — проверять ОБА файла:
`game.gd` и `game_supercell.gd`. Звук баланса (`balance_increment`) в Supercell
должен быть в `_run_balance_roll_up`, не в `_animate_win_increment` — потому что
момент начисления для игрока = приземление фишек, а не старт счётчика выигрыша.

## 2026-04-30 — Не угадывай координаты, измеряй

**Симптом:** Несколько итераций «правильного» способа центрирования
UI-элемента (бейджа над мини-рукой), ни одна не подтверждена в runtime.
Каждая новая попытка строилась поверх непроверенной гипотезы предыдущей.

**Корневая причина:** Подмена измерения предположением. После первой
неудачи продолжал предлагать «более правильный» источник координат
(HBox center → формула → first/last child rect), не проверив, что
реально возвращают эти источники в момент вызова.

**Правило:** Если правка по координатам / позиционированию / layout
не сработала с первого раза — следующий шаг не код, а диагностика.
Добавить временные `print` (или попросить пользователя их запустить),
вывести фактические rect / position / size / pivot. Гипотезы можно
выдвигать только поверх измеренных чисел.

## 2026-04-30 — `get_global_rect()` во время transform-анимации лжёт

**Симптом:** Боковые paytable-бейджи в multi-hand привязывались к
центру экрана при переключении hand_count. Ломались режимы, которые
обещал не трогать.

**Корневая причина:** `_build_paytable_badges()` → `_position_badges()`
читал `_extra_grid.get_global_rect()` ДО окончания tw_in анимации
(grid.scale = 0.15). Считывался уменьшенный rect — бейджи цеплялись
к нему.

**Правило:** Если планируется чтение `get_global_rect()` /
`get_global_position()` после программной transform-анимации
(scale/skew/rotation родителя) — `await tween.finished` ОБЯЗАТЕЛЕН
до чтения. Иначе геометрия отражает промежуточный кадр анимации,
а не финальный.

## 2026-04-30 — `queue_free` контейнера ≠ обнуление ссылок на детей

**Симптом:** Краш «Object previously freed» в `_pulse_winning_badges`
после переключения hand_count с 25 на 100.

**Корневая причина:** `_clear_paytable_badges()` делал `queue_free`
для VBox-стопок, что убивало их Label-детей. Но массив
`_badge_labels[]` оставался с прежними указателями — на следующем
DRAW цикл по нему дёргал мёртвые ноды.

**Правило:** Если ноду queue_free'ят, и её дети где-то записаны
в массивы / словари / переменные — `clear()` этих коллекций
обязателен в той же функции. Godot не зануляет такие ссылки сам.
Пройтись по всем кэшам, ссылающимся на детей удаляемой ноды.

## 2026-04-30 — Контейнер-стопка должен быть фиксированным

**Симптом:** При появлении новых бейджей в накопительной стопке
старые бейджи «перелетают» в новые координаты — визуальный шум.

**Корневая причина:** Контейнер имел `size.y = combined_minimum_size`
и `position.y = bottom_y - size.y`. Каждый новый ребёнок изменял
combined_min — позиция контейнера пересчитывалась — глобальные
координаты старых детей внутри ехали.

**Правило:** Для растущих сверху-вниз / снизу-вверх стопок —
контейнер должен быть фиксированной высоты на всю свободную зону,
а порядок задаётся через `BoxContainer.alignment` (BEGIN/END/CENTER)
+ `move_child(_, 0)` для вставки сверху. Контейнер не двигается —
двигается только free space внутри него, дети остаются на месте.

## 2026-04-30 — «Только для одного режима» через общую функцию ломает все режимы

**Симптом:** Добавление 100-hand layout сломало переключение между
3/5/10/12/25 режимами, хотя обещал «не трогать другие лейауты».

**Корневая причина:** Под флагом нового режима менял общую функцию
`_switch_hand_count` (заменил `_update_paytable_badges` на
`_build_paytable_badges`), что повлияло на все режимы, не только
на новый.

**Правило:** Когда задача звучит «не затронуть остальные режимы» —
любая правка в общей функции должна быть branch'ем под условие
нового режима, а не заменой общего пути. Default-ветка остаётся
байт-в-байт прежней. Если без изменения общего пути нельзя — это
повод явно обсудить с пользователем, не делать молча.
